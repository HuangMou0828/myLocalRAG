# Vault 向 GBrain 最佳实践对齐：V2 评估方案（供评审）

> 版本：v0.1（评估稿）  
> 日期：2026-04-25  
> 适用范围：`AIMemoryHub` 知识工作台、`wiki-vault` 生产链路、gbrain 检索链路

## 1. 背景与动机

当前系统已经形成了 `Raw Inbox -> Task Review -> Promotion -> Health` 的完整工作流，并在工程上具备可运行闭环。但随着模块持续迭代，出现了典型的“历史数据债务 + 规则漂移”问题：

- 不同阶段写入 Vault 的格式和语义约束不完全一致。
- 历史版本沉淀的数据在当前规则下可用性下降。
- 人工审核压力持续上升，吞吐受限，导致部分原始数据长期未进入有效知识层。
- gbrain 可以做强检索，但输入数据层级和质量不稳定，影响召回质量与解释性。

这次 V2 计划的核心不是“替换现有系统”，而是“在现有流水线之上增加标准化中间层”，实现：

1. 历史兼容  
2. 质量可控  
3. 迁移可回滚  
4. 与 gbrain 的最佳实践对齐

## 2. 为什么现在要做（Why Now）

### 2.1 业务层面

- 知识资产规模已进入“人工纯审核难以线性扩展”阶段。
- 团队需要将 Vault 从“记录仓库”提升为“可稳定复用的决策底座”。
- 若继续累积脏数据，后续任何检索增强（包括 gbrain）都会被输入质量上限限制。

### 2.2 技术层面

- 现有链路正向可用，但跨层 ID 与 lineage（来源追溯链）不强，影响定位和修复效率。
- Raw 到 Promotion 存在隐式衔接，用户操作后的可感知性弱。
- 页面生成、审核和巡检都在增长，规则继续堆叠会导致维护复杂度陡增。

### 2.3 风险控制窗口

- 当前有稳定的 page contract、promotion pipeline、health 巡检能力，具备“边运行边升级”的基础条件。
- 此时推进“分阶段双写迁移”，成本和风险明显低于后续再做一次全量重构。

## 3. 本次计划要解决什么问题

### 3.1 一句话目标

建立三层可演进架构：`Raw（原始） -> Normalized Atoms（标准化证据） -> Reader-first Vault（可读知识）`，并让 gbrain 优先消费后两层。

### 3.2 关键问题清单

1. 数据异构：同类知识在不同历史时期格式不一致。  
2. 脏数据累积：旧规则产物混入现行流程，质量波动大。  
3. 审核瓶颈：人工审核成本高，积压增长。  
4. 可追溯性不足：跨模块关联弱，定位源头慢。  
5. 检索噪音：低质量片段进入 embedding 后污染召回结果。  

### 3.3 预期效果

- 降低人工审核工作量，优先处理高价值候选。
- 提升检索准确率和可解释性（能清楚回答“这条结论来自哪里”）。
- 让历史数据治理从“手工救火”变为“持续收敛”。

## 4. 方案总览

### 4.1 目标架构

1. Raw 层（不可变证据层）
- 只做采集和最小清洗，不做知识结论生成。
- 保留原始上下文，作为审计与追溯基座。

2. Normalized Atom 层（标准化中间层）
- 将原始内容拆分为统一结构的“证据原子”（issue/pattern/decision/context 等）。
- 每条 Atom 带稳定主键与来源引用。
- 对 Atom 打质量分与置信度，支持分级处理。

3. Reader-first 层（对人可读层）
- 继续沿用 `projects/patterns/issues/syntheses` 结构。
- 保留 `My Notes` 等人工保护区。
- 生成逻辑从“直接吃 Raw”改为“优先吃 Atom”。

### 4.2 gbrain 对齐策略

gbrain 的 `sync/embed/query/graph-query` 仍是检索核心，但输入策略调整为：

- 默认：`Normalized + Reader-first`  
- 可选追溯：Raw（降权或按需召回）  
- 结果呈现：展示主结论 + evidence refs + lineage

这样可以同时保留召回能力与答案可信度。

### 4.3 gbrain 索引拓扑（评审前必须定稿）

为了避免 V2 上线后出现“同一证据被多层重复索引”的问题，gbrain 接入需要先明确索引拓扑，而不是在 Phase C 再边做边定。建议按下面的最小规则推进：

1. 索引单元
- Raw：保留为原始页面级索引，仅用于追溯和低优先级召回。
- Atom：作为主证据索引单元，按 `canonicalId` / `atomId` 写入，可参与 embedding。
- Reader-first：作为答案展示和导航索引单元，保留页面级索引，但不重复承载完整原始证据正文。

2. 默认检索路径
- 默认查询先命中 `Atom + Reader-first`。
- 当用户显式要求“看原文/回源/追溯”或前两层证据不足时，再召回 Raw。
- 查询结果聚合时按 `canonicalId` / lineage 去重，不允许同一证据同时以 Atom 和 Reader-first 正文片段重复占据 Top-K。

3. 去重与混排规则
- Reader-first 页面命中后，优先展示页面摘要与 evidence refs，而不是把其底层 Atom 正文再次整段重复展示。
- Atom 命中后，如果已能映射到稳定的 reader-first 页面，则在结果中提升 reader-first 的展示优先级、保留 Atom 作为证据卡片。
- Raw 命中后，仅在上层无足够证据时补充展示，并显式标记为 `raw-evidence`。

4. 存储与切流原则
- Phase B 双写期间，V2 feed 必须可独立构建与 dry-run，不影响现有 repo 级同步。
- Phase C 之前要明确：是继续让 gbrain 直接同步仓库文件，还是新增面向 Atom/Reader-first 的专用 feed 目录或导出产物。
- 无论采用哪种接入方式，都必须保证“同一证据只有一个主索引身份”，其余层通过 lineage 关联而不是重复建主记录。

## 5. 数据治理设计

### 5.1 V2 数据契约（建议最小集）

- `canonicalId`：全局稳定标识  
- `kind`：issue/pattern/decision/context/question 等  
- `title`  
- `summary`  
- `topics`  
- `sourceRefs`  
- `schemaVersion`（V2）  
- `qualityScore`（0-100）  
- `confidence`（0-1）  
- `updatedAt`

补充说明：上面只是 Atom 层最小字段，不等于可直接替代当前 reader-first 页面契约。V2 必须显式定义“Atom 合约”和“Reader-first 合约”的映射关系。

### 5.1.1 与现有 page contract 的兼容映射

当前 reader-first 页面仍依赖既有 contract，至少包括：

- `title`
- `type`
- `schemaVersion`
- `status`
- `project`
- `evidenceCount`
- `updatedAt`
- `promotionState`
- `approvedAt`

因此 V2 评估阶段必须同步产出一份兼容映射表，明确：

- Atom 的 `kind` 如何映射到 reader-first 的 `type`
- Atom 聚类后的 `project/topics/sourceRefs` 如何映射到页面级 `project/evidenceCount`
- 人工审核产生的 `promotionState/approvedAt` 挂在哪一层，并如何回写到 reader-first 页面
- `schemaVersion` 如何从当前 reader-first `schemaVersion: "1"` 渐进升级，而不是隐式替换

原则上：

- Atom 层负责证据标准化与质量评分
- Reader-first 层继续负责页面契约、人工保护区和可读输出
- Promotion/Health/generator 在切流前都必须有一张明确的字段对照表，而不是运行时猜测

### 5.1.2 稳定 ID 与 lineage 规则

V2 如果没有确定的身份规则，双写、一致性校验、回滚和历史迁移都会失真。建议在 Phase A 就固定以下身份模型：

- `rawId`：Raw Inbox 或 source 页面稳定标识，来源于现有入库主键或稳定路径 hash
- `atomId`：单条标准化证据原子 ID，由 `rawId + normalized evidence span + kind` 稳定生成
- `canonicalId`：跨版本、跨修订的语义稳定标识，用于聚类与去重
- `pageId`：reader-first 页面稳定标识，对应当前 `issues/`、`patterns/`、`projects/`、`syntheses/` 页面
- `lineage`：至少记录 `rawId -> atomId -> canonicalId -> pageId` 的关联链

补充约束：

- 历史迁移数据若无法直接生成稳定 `canonicalId`，必须保留 alias / legacyId 映射表，不能只靠标题相似度临时合并
- 页面重命名、聚类拆分、聚类合并都要保留 lineage 事件，避免 approval/history 丢失
- gbrain 检索层与应用层展示层都要消费同一套身份字段，不能各自维护一套去重键

### 5.2 脏数据分级

- `clean`：结构完整、可直接用于生成与检索  
- `suspect`：结构可用但置信不足，进入人工复核队列  
- `legacy`：历史格式/字段缺失严重，降权且纳入迁移队列  

### 5.3 处理策略

- 默认不硬删除原始数据。  
- 通过“降权 + 软归档 + 待修复队列”治理。  
- 所有修复动作可追踪且可回滚。  

### 5.4 Raw -> Atom 自动化责任边界（强制）

为避免“积压只是换位置”的问题，V2 必须明确以下边界：

1. Raw -> Atom 必须自动化
- Raw 到 Atom 的生成、字段归一、质量评分由系统自动执行（LLM + 规则），禁止人工逐条生成 Atom。
- 人工角色是“审核决策者”，不是“Atom 生产者”。

2. 质量分流规则（默认）
- `qualityScore 高`：仅对低风险类型（建议 `issue/pattern`）允许自动晋升，且必须满足门禁（lineage 完整、sourceRefs 完整、非重复内容）。
- `qualityScore 中`：进入人工复核队列，人工仅做确认/驳回，不要求重写 Atom。
- `qualityScore 低`：软归档或降权，不进入日常人工队列，仅进入治理批次抽样修复。

3. 自动化覆盖率门禁
- `Raw -> Atom 自动化覆盖率` = 自动生成 Atom 的 Raw 条数 / 进入管道的 Raw 总条数。
- 该指标若低于 95%，不进入下一阶段切流。

4. 人工效率口径（用于“20%+”目标）
- 不再使用“Raw 审核时长”作为主指标。
- 主指标改为：`每条成功入 Vault 知识的人审分钟数 = 总人工审核分钟 / 成功入 Vault 条数`。
- Phase C 的“下降 20%+”指该口径相对 Phase A 基线下降。

## 6. 分阶段计划（建议）

### Phase A：标准定义与评估基线（1-1.5 周）

目标：先统一规则，再动生产路径。

产物：
- Data Contract V2 文档
- 字段映射表（旧 md -> V2 Atom）
- 质量评分规则（结构完整度、重复度、冲突度、时效性）
- gbrain 索引拓扑说明（索引单元、存储位置、去重规则、默认混排路径）
- 稳定 ID / lineage 规则与 alias 迁移策略
- 当前 page contract 到 V2 的兼容映射表
- 冻结评估样本集、查询集、统计口径
- dry-run 报告模板

验收：
- 抽样集（建议 >= 200 条）完成自动分类与人工复核，且保留标注结果。
- 至少冻结一组查询评估集（建议 >= 50 个真实查询），覆盖 issue/pattern/project/synthesis 四类检索意图。
- 每个指标都有明确计算口径、脚本入口和基线快照，可重复执行。
- 评分规则、ID 规则、兼容映射达到可解释、可复现。

### Phase B：双写迁移（1-2 周）

目标：不切老路径，先并行写新层。

产物：
- 现有流程继续写入原有结构
- 同步生成 Atom + lineage
- 质量分级与审核队列联通

验收：
- 双写一致性通过：抽样集上 reader-first 页面关键字段一致率达到预设阈值（建议 >= 99%）
- lineage 完整率达到预设阈值（建议 >= 99.5%）
- 双写期间不新增 `My Notes` 覆盖或 approval 元数据丢失问题
- 用户核心流程无感中断，现有 Promotion / Health / publish 主链路可正常运行
- `Raw -> Atom 自动化覆盖率 >= 95%`，且人工不承担逐条 Atom 生成

### Phase C：读路径切换（约 1 周）

目标：让审核和检索优先消费 V2 产物。

产物：
- Promotion/Health 读取 V2 视图
- gbrain 同步与 embedding 优先吃 V2 feed

验收：
- 在冻结查询集上，V2 的 Top-K 命中率显著不低于旧链路基线，且无效召回率达到预设下降目标
- 答案可追溯率达到预设阈值（建议 >= 95% 查询结果可落到 evidence refs + lineage）
- 每条成功入 Vault 知识的人审分钟数较旧链路下降到预设目标区间（例如下降 20% 以上）
- 若任一核心指标劣化超过阈值，必须可通过回退开关切回旧读路径

### Phase D：历史修复与持续治理（持续）

目标：逐步降低 legacy 占比。

产物：
- 周期性健康报告
- 异常队列修复机制
- 回归检测策略

验收：
- 以周为单位输出固定口径报表，至少包含 legacy 占比、疑似重复占比、lineage 缺失占比
- legacy 占比按预设节奏持续下降
- 高优问题闭环时延持续缩短，且没有通过“简单隐藏问题”换取指标改善

## 7. 风险评估与缓解

### 7.1 主要风险

1. 规则过严导致召回不足  
2. 规则过松导致噪音回流  
3. 双写期字段漂移造成统计混乱  
4. 迁移期间团队理解不一致（“看起来都能跑，但口径不一致”）  
5. 历史数据量大，修复周期长于预期  

### 7.2 缓解措施

- 设置质量阈值开关，可按仓/目录渐进放量。  
- 保留旧读路径回退开关，切流失败可快速回滚。  
- 每阶段都建立固定抽检与复盘节奏（周维度）。  
- 统一 lineage 字段，保证每条结论能追到 source。  
- 优先治理高频主题与高价值数据域，避免平均用力。  

## 8. 成功标准（建议 KPI）

### 8.1 质量指标

- `clean` 占比
- 字段完整率
- 重复率/冲突率
- lineage 完整率
- V2 -> reader-first 字段映射成功率

### 8.2 效率指标

- 每条成功入 Vault 知识的人审分钟数
- 待审核积压量
- 从 Raw 到可检索知识的中位时延

### 8.3 检索指标

- Top-K 命中率
- 无效召回率
- 答案可追溯率（带 evidence refs）
- 去重后有效结果占比
- Raw 兜底召回占比（用于观察上层质量是否足够）

### 8.4 稳定性指标

- 回滚演练成功率
- `My Notes` 等保护区保留率（应接近 100%）
- approval / promotion 元数据保留率
- 双写一致性失败率

## 9. 评审时建议重点讨论的决策点

1. V2 初始策略是“准确优先”还是“覆盖优先”？  
2. `legacy` 数据默认“降权可见”还是“仅追溯可见”？  
3. 双写窗口定 1 周还是 2 周？  
4. 首批治理范围按“高价值项目”还是“全量均摊”？  
5. gbrain 在 V2 中继续直接索引 repo 文件，还是切到专用 V2 feed？  
6. 身份主键以 `rawId/atomId/canonicalId/pageId` 哪一层作为检索去重主键？  

## 10. 评估方法补充（建议写进执行稿）

为了避免评估变成主观判断，建议在执行前固定以下方法：

1. 基线冻结
- 在 Phase A 开始时冻结一份旧链路基线，包括查询集、样本集、Top-K 结果快照和人工审核耗时样本。

2. 查询集设计
- 查询集应覆盖：
  - issue 检索（症状/报错/修复）
  - pattern 检索（最佳实践/推荐做法）
  - project 检索（模块/仓库/结构导航）
  - synthesis 检索（规划/比较/结论）

3. 统计口径
- Top-K 命中率：答案是否在前 K 个可用结果中出现
- 无效召回率：返回结果相关性低或无法支撑答案的比例
- 可追溯率：结果是否可落到 evidence refs + lineage
- 人工效率：`总人工审核分钟 / 成功入 Vault 条数`（按周统计并与 Phase A 基线对比）

4. 放量条件
- 未达到阶段阈值时，不进入下一阶段
- 指标劣化但原因未定位时，不允许以“先上线再修”替代评审结论

## 11. 不做这次计划的代价（机会成本）

- 审核吞吐会继续受人工瓶颈限制。  
- 脏数据继续进入检索链路，导致“看起来有数据，实际不可用”。  
- 后续再重构时迁移成本更高、风险更大。  
- 团队对知识库可信度下降，影响复用与决策效率。  

## 12. 结论

本计划建议作为“结构性治理项目”启动评估。  
它不是重写系统，而是在现有可用链路上加一层标准化与质量控制，目标是在保持业务连续性的前提下，逐步把 Vault 和 gbrain 联动提升到可长期演进的状态。
