# V2-only 新 Vault 日常操作规范

> 日期：2026-04-26
> 状态：执行标准
> 适用范围：知识工作台、Vault 审核流程、日常知识沉淀
> 目标：让新 Vault 只通过一条稳定主链生产，避免回到旧的多入口、多语义状态

## 1. 一句话原则

新 Vault 只走一条默认主路：

`知识采集 -> 系统候选 -> 人工 approve -> reader-first Vault -> 健康巡检`

任何偏离这条链路的能力，都只能算辅助工具，不能算默认生产流程。

## 2. 角色分工

### 2.1 `知识采集`

职责：

- 接住原始材料
- 补最小上下文
- 判断是否值得送审

允许的内容：

- capture
- note
- document
- OpenClaw 导入条目
- 临时结论
- 半成品观察

不要求：

- 一开始就结构完整
- 一开始就能直接变成正式知识页

### 2.2 `升格审核`

职责：

- 审核系统整理好的候选
- 决定是否值得进入长期知识层

默认动作：

- `Approve`
- `Dismiss`
- `Preview`

这里是新 Vault 的唯一默认审核入口。

### 2.3 `健康巡检`

职责：

- 检查最终知识层结构是否稳定
- 发现断链、孤儿页、弱摘要、项目空洞

注意：

- 健康巡检不负责生产知识
- 健康巡检只负责发现问题和把你带回正确入口

## 3. 日常主流程

### 3.1 原料进入

1. 在 `知识采集` 新建、导入或同步原始条目
2. 尽量补齐这些字段：
- 标题
- 内容
- 项目
- topic
- intake stage
- confidence

最低要求：

- 标题能让人看懂
- 内容里至少有一个明确问题、结论或经验点

### 3.2 送审标准

只有满足下面至少一类，才建议送 `升格审核`：

- 有明确问题与处理思路
- 有可复用做法或 pattern
- 有可以独立成立的结论页雏形
- 和某个项目的长期知识明显相关

不建议送审的情况：

- 只是聊天过程噪音
- 只有一句模糊结论
- 没有足够上下文，自己过几天也看不懂
- 明显只是临时过程记录

### 3.3 进入候选池

操作规则：

1. 把 `intakeStage` 设成 `wiki-candidate`
2. 点 `保存并送升格审核`
3. 默认去 `升格审核`
4. 在 `Issue / Pattern / Synthesis` 三类候选里继续判断

说明：

- 不再要求先经过 `任务筛选`
- `任务筛选` 不是默认主路

### 3.4 审核动作

#### `Approve`

适用条件：

- 标题已经稳定
- 摘要已经能表达核心价值
- 证据链足够支撑这个页面存在
- 它值得长期留在 Obsidian 里

效果：

- 写入 reader-first Vault
- 进入 `issues/`、`patterns/`、`syntheses/`、`projects/` 等最终层

#### `Dismiss`

适用条件：

- 价值不足
- 重复
- 上下文不够
- 暂时不值得进入长期知识层

效果：

- 候选从默认审核视图移出
- 不进入最终知识层

#### `Preview`

适用条件：

- 不确定写入后的页面质量
- 想先看生成内容是否像正式笔记

默认建议：

- 对边界候选先 `Preview`
- 对高质量、结构完整的候选可直接 `Approve`

## 4. Approve 的最低标准

### 4.1 Issue

至少应当具备：

- 问题主题清晰
- 有 symptom / cause / fix / validation 中的多数信息
- 不是单纯情绪化报错记录

### 4.2 Pattern

至少应当具备：

- 可复用，不只是一次性解法
- 有“什么时候用”
- 有“推荐形状”
- 最好带 tradeoff

### 4.3 Synthesis

至少应当具备：

- 标题本身能成立
- 摘要已经像结论，不像聊天摘抄
- 放进 Obsidian 后别人能单独阅读

### 4.4 Project

项目页不是直接审批对象，但会由已批准证据聚合生成。

因此项目页成立的前提是：

- 至少已经有值得保留的 issue / pattern / synthesis
- 项目名称或边界是稳定的

## 5. 什么不能进入新 Vault

以下内容默认不进入最终层：

- 未 approve 的候选
- 未批准的 source evidence
- 未批准的 knowledge evidence
- 单纯检索噪音
- 没有独立阅读价值的过程碎片

一句话判断：

如果这页放进 Obsidian 后，你自己都不想再次点开，它就不该进新 Vault。

## 6. Health 的使用规则

健康巡检发现问题后，处理顺序如下：

1. 如果是结构问题：
- 先修链接
- 先补锚点
- 先补项目关联

2. 如果是知识质量问题：
- 回到 `升格审核`
- 重新判断是否值得留

3. 如果是证据不足问题：
- 回源补上下文
- 再重新进入 `升格审核`

健康巡检不是为了制造更多候选，而是为了保证最终层长期可维护。

## 7. 每个目录的业务含义

- `vault/sources/`
  证据层，允许粗糙，不等于正式知识

- `vault/inbox/`
  候选层、追溯层、过渡层

- `vault/issues/`
  已批准的问题页

- `vault/patterns/`
  已批准的可复用模式页

- `vault/syntheses/`
  已批准的综合结论页

- `vault/projects/`
  由已批准知识聚合出的项目页

- `vault/.gbrain-v2-feed/`
  检索数据包，不是人读主页面

## 8. 操作禁令

以下行为不再属于标准做法：

- 把 `sources/*.md` 当最终知识页
- 未审核就默认写入 `issues/` 或 `patterns/`
- 把旧 `promotion-queue` 当主审核视图
- 依赖文件名携带技术状态
- 把历史兼容入口误当成当前主流程

## 9. 每周最小维护动作

建议每周至少做一次：

1. 清一批 `升格审核` 候选
2. 看一次 `健康巡检`
3. 检查是否有明显噪音再次流入候选池
4. 抽查 3 到 5 个 reader-first 页面

抽查重点：

- 标题是否像笔记
- 摘要是否独立可读
- Evidence 是否够支撑
- 项目关联是否稳定

## 10. 当前系统执行口径

这份规范对应当前已经落地的实现口径：

- 新 Vault 默认只走 V2
- reader-first 页面只由已批准证据重建
- 最终页状态应表达正式知识，而不是候选态
- `任务筛选` 不再是默认主流程入口
